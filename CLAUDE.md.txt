# Welcome, Baby — Prototype Build Guide

## Project context
"Welcome, Baby" is a new-employee-parent experience on the DBS intranet for
employees who've just had a newborn. This repo is a **front-end-only working
prototype** that will be handed to developers to rebuild on **Liferay**. The
developers will wire up real backend APIs and CMS connections; nothing in this
repo needs to be production-grade or framework-based.

## Stack & constraints (non-negotiable)
- **Vanilla HTML, CSS, and JavaScript only.** No frameworks (no React, Vue,
  Svelte, jQuery, Tailwind, Bootstrap, etc.).
- No build step. The prototype must run by double-clicking `index.html` or
  serving the folder with a simple static server.
- No npm dependencies in the runtime output. Dev tools (e.g. a formatter) are
  fine, but the deliverable must be plain files.
- **Responsive from 1024px and up only.** Do not design or test for tablet or
  mobile breakpoints. Use a fluid layout that scales gracefully up to ~1920px.
- Browser target: latest Chrome, Edge, Safari, Firefox. No IE.
- Accessibility: semantic HTML, alt text on images, visible focus states,
  WCAG AA color contrast. Keyboard-navigable.

## Visual direction
- **Match the Figma visually.** I will describe screens in chat — colors,
  spacing, typography, components — and you implement to spec.
- When I describe a color, store it as a CSS custom property in
  `assets/css/tokens.css` so devs can swap to DLS tokens later.
- Use a single source of truth for spacing, radii, type sizes, and colors —
  all as CSS variables in `tokens.css`.
- Default font stack until I say otherwise: `system-ui, -apple-system,
  "Segoe UI", Roboto, sans-serif`.

## Code style
- Indent: tabs (or 2 spaces — pick one and stay consistent).
- Class names: kebab-case BEM-ish (`.card`, `.card__title`, `.card--featured`).
- One stylesheet per concern (`tokens.css`, `styles.css`); avoid inline styles
  except for prototype-only state demos.
- JavaScript: vanilla ES modules, no transpilation. Use `data-*` attributes
  for behavior hooks, not classes.
- Comment any non-obvious logic with a one-liner explaining intent for the
  Liferay devs.

## Navigation behavior
- The prototype is **clickable between screens / states**. Use one of:
  - Multiple `.html` files in `/screens/` linked by anchor tags, OR
  - A single-page approach with `hidden` attribute toggled by JS on
    `data-screen` containers.
- Pick the simpler option per screen count. Ask me before introducing
  routing libraries (you shouldn't need any).

## Liferay handoff rules
The developers will port this to Liferay templates (Freemarker/JSP) and
fragments. To make their job easy:
- Keep markup semantic and free of prototype-only wrappers.
- Avoid clever JS patterns; prefer dumb, readable code.
- Anywhere content will be CMS-managed, wrap it in a comment like
  `<!-- CMS: hero headline -->`.
- Anywhere a backend call will happen, stub it as a function in `app.js` with
  a `// TODO(api): ...` comment describing the expected endpoint and payload.

## Workflow rules for Claude Code

For **any new feature, screen, change, or refactor**, you MUST follow this
three-phase loop. Do not skip phases. Do not write or modify code until
Phase 3.

### <Plan>
- Break down the request into clear, sequential steps.
- Anticipate common rookie mistakes (e.g. building everything at once,
  hardcoding values that belong in tokens.css, duplicating components,
  skipping responsive checks, forgetting CMS/API stub comments).
- Include a checklist of what to tackle in order.
- Note any assumptions you're making and any clarifying questions for me.

### <Review>
- Show me the full plan before executing.
- Ask if I want to adjust, remove, reorder, or add steps.
- Wait for my explicit confirmation ("go", "execute", "approved", etc.)
  before moving on. Silence is not consent.

### <Execute>
- After I confirm, implement **one step at a time**.
- Show progress incrementally — do not dump the entire output at once.
- At the end of each step:
  1. Summarize what was done (files touched, key decisions).
  2. State what's next.
  3. Pause and wait for me to say continue, unless I've pre-approved
     running all steps back-to-back.

### Always-on rules
- Always read CLAUDE.md and the existing files before planning, so the
  plan respects current tokens, components, and structure.
- Reuse existing components and tokens whenever possible — call this
  out in the plan.
- Never delete or rename files without confirming in the Review phase.
- Never add features, copy, screens, or dependencies I didn't request.
- After all steps are done, give me a one-line "how to view" instruction.

### When this loop does NOT apply
- Pure questions ("how does X work?", "what does this file do?")
- Reading or summarizing files
- Trivial typo fixes I've already pointed at a specific line for
- Bug reports — use the Bug-fix workflow instead
- Any request prefixed with `[no-loop]`

**Default behavior on any ambiguous request: assume the loop applies.**
**DO NOT EXECUTE CODE YET — wait for Review approval.**

## Bug-fix workflow

When I report a bug, defect, or "this isn't working" — even if I'm casual
about it — you MUST follow this three-phase loop. Do not patch code until
Phase 3.

### <Debug>
- Identify which component / section / file is affected.
- Determine if it's a **visual issue** (CSS, layout, tokens, responsiveness)
  or a **functional issue** (JS logic, event handlers, navigation, state).
  If both, say so.
- List possible causes based on my description, ranked most → least likely.
- Reference specific lines or selectors when you can — vague diagnoses
  don't count.
- Create a step-by-step fix strategy that targets the root cause, not just
  the symptom.

### <Review>
- Show me your diagnosis and proposed fix before implementing.
- Let me confirm this is the correct issue.
- Ask clarifying questions if my bug description was unclear (e.g.
  "which browser?", "what did you expect to see vs what you saw?",
  "at what viewport width?").
- Wait for my explicit confirmation before moving on.

### <Execute>
- Implement the fix for **this specific bug only**.
- Do not refactor unrelated code, "improve" nearby logic, or sneak in
  drive-by tweaks. If you spot something else worth fixing, mention it
  AFTER the fix and ask whether to address it separately.
- After the fix, verify:
  1. The reported bug is resolved (describe how you confirmed).
  2. The fix did not break adjacent functionality — name the screens,
     components, or interactions you checked.
- Summarize: files touched, what changed, what was tested, anything you
  couldn't verify without me clicking through.

### Bug-fix specific rules
- Never assume the bug is "fixed" just because the code now looks right.
  State what you actually verified.
- If the bug can't be reproduced from my description alone, say so in
  Review and ask for: steps to reproduce, viewport width, browser, and
  a screenshot if visual.
- If the root cause is in tokens.css or a shared component, flag that the
  fix may affect other screens and list them in Review.
- If the same bug pattern could exist elsewhere in the prototype (e.g.
  same broken handler copy-pasted), call it out in Review — but still
  only fix what I confirm.

**DO NOT EXECUTE CODE YET — wait for Review approval.**

## Out of scope
- Backend, auth, real data
- Mobile / tablet layouts
- Build tooling, bundlers, CSS preprocessors
- Analytics, tracking, cookie banners
- Animations beyond simple CSS transitions (ask before adding)
